摘要

现代操作系统内核（如Linux）使用eBPF子系统来允许用户空间扩展内核功能。为确保安全性，内核验证器对这些扩展进行静态分析；然而，其不精确的分析经常导致错误拒绝安全扩展，暴露了验证器精度和计算复杂性之间的关键张力，限制了内核可扩展性。

作者提出了一种证明引导的抽象精化技术，显著提高了验证器的精度，同时保持低内核空间复杂性。关键见解是将复杂推理解耦到用户空间，同时通过形式化证明弥合差距。当遇到不确定性时，验证器启动抽象精化过程而不是拒绝扩展。

评估结果显示，该技术使验证器接受了512个真实世界eBPF程序中的403个，这些程序之前被错误拒绝，为更可靠和灵活的内核扩展铺平了道路。

背景与动机

操作系统内核设计为高可扩展性，以适应不同的用户空间工作负载。eBPF子系统允许用户空间通过扩展向内核注入专门功能，支持各种任务，包括性能分析、安全监控、文件系统管理和设备驱动程序增强。

eBPF验证器

eBPF扩展在内核空间执行但由不受信任的第三方开发，因此内核必须严格确保其安全执行。这是通过内核内静态分析器（称为验证器）实现的，它利用抽象解释技术。验证器系统地探索所有潜在的控制流路径，并通过抽象语义解释每条指令，验证关键属性，如内存安全和正确的内核交互。

eBPF验证过程

0: r1 = map_lookup(...) // r1 = ptr(size=16)
1: r2 &= 0xf          // r2 ∈ [0,15]
2: r2 <<= 1           // r2 ∈ [0,30]
3: r1 += r2           // r1.off ∈ [0,30]
4: r0 = *((u8 *)r1)   // unsafe: off+access_sz > mem_sz

代码清单1：正确拒绝不安全扩展。注释说明了验证器的分析过程。

验证器使用高效的抽象域来近似每个变量的潜在值，包括四个不同的区间域（考虑不同位大小和符号表示）和一个三态域（跟踪单个位的状态）。这些抽象使得能够推导变量范围。

问题与挑战

尽管eBPF验证器提供了强大的安全保证和扩展的高性能，但设计精确的内核内验证器面临重大挑战，特别是在平衡分析精度与系统复杂性方面。

精度与复杂性的权衡

提高精度通常需要纳入复杂的静态分析技术。然而，内核环境对验证器复杂性和运行时开销施加了严格约束。增加验证器复杂性不仅扩大了潜在攻击面，引入安全漏洞，还导致运行时开销，影响系统延迟并消耗内核资源，最终降低用户空间应用程序性能。

精度限制示例

0: r1 &= 1          // r1 = 0000x (x表示未知)
1: r2 = r1          // r2 = 0000x
2: r2 <<= 1         // r2 = 000x0
3: if r2 > 1 goto exit // fallthrough, r2 <= 0001 (因此x=0)
4: r1 += 8          // r1 ∈ [0,8] (但r1必须为0)
5: load_bytes(..., r1); // unsafe (错误拒绝)

代码清单2：安全扩展的错误拒绝。括号内是我们的注释。

在实践中，eBPF扩展比这些简化示例更复杂，它们部署在各种场景中实现丰富功能。因此，验证器的固有精度不足已成为显著瓶颈。

精度限制
  • 处理算术运算时丢失重要信息
  • 精确计算被局部更新的过近似替代
  • 复杂计算和变量间关系跟踪不足
实际影响
  • 许多安全扩展被错误拒绝
  • 开发者需要大量手动工作解决验证器反馈
  • 在某些情况下缓解不可行，限制内核可扩展性

设计与实现

作者提出了BCF（eBPF证书框架），一种证明引导的抽象精化技术，应用于增强eBPF验证器。该方法通过将复杂推理委托给用户空间，同时在内核空间进行高效证明检查，实现了高精度和低开销的平衡。

设计实现：
实现方法:

BCF扩展了eBPF验证器，添加了向后分析、符号跟踪和证明检查功能。系统调用接口仅需五个额外字段，无需新的系统调用或命令。

技术选择:

采用独立的二进制编码格式，基于紧凑的u32方案。操作符编码在第一个四字节单元中，封装操作、长度和类型信息。

优化策略:

符号跟踪集成到分析阶段，仅在精化时启用，确保标准分析不受影响。证明检查器支持45个原始规则，包括布尔、等式和位向量理论的标准化规则。

关键设计原则

保持验证器简单

最大化验证器的分析使用，同时保持其效率。当验证器无法继续时，表示需要抽象精化而不是拒绝扩展。

抽象精化

将计算密集型推理卸载到用户空间，验证器仅描述精化任务，复杂推理在用户空间执行。

机器可检查证明

要求机器可检查的证明来认证精化的有效性，仅在证明检查后采用精化。

BCF工作流程

1

初始抽象解释

验证器执行其高效的抽象解释，当检测到潜在错误时触发精化过程。

2

向后分析和符号跟踪

使用向后分析确定启动符号跟踪的适当程序位置，收集相关寄存器的符号表达式。

3

精化抽象

推导更精确的抽象，使分析能够继续进行，并使用符号信息捕获精化的正确性作为形式条件。

4

用户空间证明生成

条件委托给用户空间，使用SMT求解器生成证明条件的有效性。

5

内核空间证明检查

收到证明后执行严格的证明检查，验证每个步骤是否符合规定的规则。

测试与评估

作者使用一组真实世界的eBPF程序评估BCF，这些程序尽管安全，但被当前验证器错误拒绝。评估结果显示BCF在精度改进和开销方面表现优异。

关键发现
  • 精度改进: 通过集成BCF，验证器成功加载了512个eBPF程序中的403个，接受率达到78.7%
  • 高效证明检查: 平均证明大小限制在541字节，证明检查过程平均仅需48.5微秒
  • 低开销: 总分析时间平均为9.0秒，证明检查阶段占总时间不到1%，用户空间组件平均占21%

结论

作者提出了BCF，一种证明引导的抽象精化技术，显著提高了eBPF验证器的精度，同时保持低内核空间复杂性。通过将复杂推理解耦到用户空间，同时通过形式化证明弥合差距，BCF解决了验证器精度和计算复杂性之间的关键张力。
