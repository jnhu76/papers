摘要

作者提出了Gorilla，一个基于LLaMA微调的模型，在编写API调用方面超越了GPT-4的性能。当与文档检索器结合时，Gorilla展示了强大的适应测试时文档变化的能力，支持灵活的用户更新或版本更改。它显著缓解了直接提示LLM时常见的幻觉问题。

为了评估模型的能力，作者引入了APIBench，一个包含HuggingFace、TorchHub和TensorHub API的综合数据集。检索系统与Gorilla的成功集成展示了LLM更准确使用工具、跟上频繁更新的文档以及提高输出可靠性和适用性的潜力。

背景与动机

近年来，大语言模型在自然语言对话、数学推理和程序合成等方面取得了显著进展。然而，尽管有这些进步，LLM仍然受到固定权重集存储信息和有限上下文计算的限制。此外，随着世界的变化，LLM需要重新训练以更新其知识和推理能力。

通过赋予LLM使用工具的能力，可以为它们提供访问更大和变化的知识库，并完成复杂的计算任务。通过提供对搜索技术和数据库的访问，可以增强LLM处理更大、更动态知识空间的能力。同样，通过提供对计算工具的访问，LLM可以完成复杂的计算任务。

从少量手动编码工具到调用大量变化的云API的转变，可以将LLM转变为计算基础设施和网络的主要接口。从预订整个假期到举办会议的任务，都可以简化为与具有航班、租车、酒店、餐饮和娱乐网络API访问权限的LLM对话。

问题与挑战

支持可能包含数百万个变化API的网络规模集合需要重新思考集成工具的方法。不再可能在单个上下文中描述完整的API集合。许多API将具有重叠功能，并带有细微的限制和约束。在这个新环境中简单地评估LLM需要新的基准。

主要挑战
  • API规模与变化性: 支持数百万个不断变化的API
  • 功能重叠: 许多API具有重叠功能，但带有细微限制
  • 文档更新频率: API文档更新速度超过LLM重新训练频率
  • 幻觉问题: LLM容易生成不存在或错误的API调用
  • 约束理解: API调用通常带有参数大小、精度等约束

设计与实现

APIBench数据集

作者构建了APIBench，一个从公共模型中心抓取ML API的大型API语料库。选择了三个主要模型中心进行数据集构建：

TorchHub

94个API调用

TensorHub

696个API调用

HuggingFace

925个精选模型

每个API条目转换为包含以下字段的JSON对象：{domain, framework, functionality, api_name, api_call, api_arguments, environment_requirements, example_code, performance, description}。

Gorilla模型

Gorilla是基于LLaMA-7B的检索感知微调模型，专门用于API调用。作者使用自指导方法生成{instruction, API}对，并将其转换为用户-代理聊天式对话进行微调。

检索感知训练

对于带有检索器的训练，指令调优数据集在用户提示后附加了额外的"Use this API documentation for reference: "。通过这种方式，作者旨在教导LLM解析问题的后半部分来回答前半部分。

API验证方法

作者采用AST子树匹配策略来评估生成的API的功能正确性。将生成的代码解析为AST树，然后找到根节点为API调用的子树，并使用它来索引数据集。

测试与评估

基线模型

作者将Gorilla与以下最先进的语言模型在零样本设置下进行比较：

  • GPT-4: OpenAI的最新模型
  • GPT-3.5-turbo: 专门为对话设计的RLHF调优模型
  • Claude: Anthropic的语言模型，以其长上下文能力著称
  • LLaMA-7B: Meta的大型语言模型，最优秀的开源模型

检索器设置

评估了四种不同的检索器设置：

零样本 (0-shot)

不使用检索器，仅使用用户的自然语言提示

BM25

使用BM25检索器获取最相关的API文档

GPT-Index

使用GPT-Index检索模型获取相关文档

Oracle

100%返回相关文档，表示性能上限

主要结果

关键发现
  • 轻量微调的Gorilla在零样本设置下获得了最先进的性能，比GPT-4高20.43%，比ChatGPT高10.75%
  • 与LLaMA相比，改进高达83%，表明微调优于检索
  • 在测试时添加非最优检索器有时会误导模型并导致更多错误
  • 在微调管道中纳入真实检索器获得了显著更好的结果，在Torch Hub中比无检索器训练高12.37%，在HuggingFace中高23.46%

测试时文档变化适应

API文档的快速演变性质对LLM在该领域的应用构成了重大挑战。这些文档的更新频率通常超过LLM的重新训练或微调计划，使得这些模型特别脆弱于其设计处理信息的变化。

通过引入Gorilla的检索感知训练，可以轻松适应API文档的变化。这种新颖的方法使模型即使在依赖的API文档经历修改时也能保持更新和相关性。

约束理解能力

作者评估了Gorilla在理解API调用约束方面的能力。对于任何给定任务，调用哪个API通常是多种因素之间的权衡。在ML API的范围内，期望Gorilla尊重诸如准确性、模型中可学习参数数量、磁盘大小、峰值内存消耗、FLOPS等约束。

约束理解示例

用户提示："调用一个使用少于10M参数但保持ImageNet准确率至少70%的图像分类模型"

这样的提示对LLM准确解释和响应提出了重大挑战。LLM不仅必须理解用户的功能描述，还需要推理请求中嵌入的各种约束。

结论

LLM正在迅速在不同领域获得普及。在这项研究中，作者重点介绍了旨在增强LLM准确识别特定任务适当API能力的技术——这是这项技术发展中一个重要但经常被忽视的方面。

由于API作为一种通用语言，使不同系统能够有效通信，它们的正确使用可以提升LLM与更广泛世界工具交互的能力。在本文中，作者提出了Gorilla，一种用于微调LLM调用API的新颖管道。微调模型的性能在作者收集的三个大规模数据集中超过了提示最先进的LLM（GPT-4）。

Gorilla生成可靠的ML模型API调用而无幻觉，展示了令人印象深刻的适应测试时API使用变化的能力，并可以在选择API时满足约束。

未来方向

  • 扩展到更多领域和API类型
  • 改进检索器性能以进一步提高准确性
  • 探索更复杂的约束理解和推理能力
  • 研究API调用序列和组合的可能性
