论文概览

FastLanes 是一个开源项目，旨在通过重新设计压缩布局来加速大数据格式（如Parquet、ORC和列式数据库格式）的解码性能。

核心创新：通过两种主要思想显著加速所有常见轻量级压缩方案（DICT、FOR、DELTA和RLE）的解码：

  1. 针对虚拟1024位SIMD寄存器，泛化位打包基本操作中的值交错技术
  2. 在所有列中以相同的统一转置布局重新排序元组，实现最大独立工作

解决的问题

  • 硬件多样性带来的软件开发挑战
  • SIMD指令集的技术债务
  • 解码性能瓶颈

关键技术

  • 虚拟1024位SIMD寄存器
  • 统一转置布局
  • 标量代码自动向量化

核心问题

如何在异构硬件环境下实现高效、可移植的轻量级压缩数据解码，同时避免SIMD指令集带来的技术债务？

现有方法的局限性

硬件多样性挑战

  • SIMD宽度不断增加（64位→128位→256位→512位）
  • 多种指令集架构（x86, ARM, POWER, RISC-V）
  • SIMD指令子集不兼容

技术债务问题

  • 平台特定的SIMD内部函数
  • 代码维护复杂
  • 未来硬件兼容性差

真实世界痛点

  • 数据分析系统中压缩数据解码成为性能瓶颈
  • 云环境中硬件平台多样化
  • 维护多平台优化代码成本高昂
  • 顺序依赖（DELTA、RLE）限制数据并行性

核心贡献（4C模型）

Concept（新概念）

  • 虚拟1024位SIMD寄存器（FLMM1024） - 抽象硬件指令集
  • 统一转置布局 - 适用于所有车道宽度的数据重排方案
  • 标量友好的数据并行性 - 通过数据布局而非指令实现并行
  • FastLanes-RLE - 基于DELTA和DICT的新型RLE编码

Challenge（解决的难题）

  • SIMD宽度多样性 - 25年间SIMD宽度增加了8倍
  • 指令集异构性 - x86、ARM、POWER、RISC-V多平台支持
  • 解码依赖关系 - DELTA和RLE编码中的顺序依赖
  • 代码可移植性 - 避免平台特定的SIMD内部函数

Construction（方法设计）

  • 1024位交错位打包 - 值循环分布到128个8位车道
  • 统一转置布局 - "04261537"顺序组织8×16转置块
  • 简单指令集设计 - 仅使用基本操作
  • 标量代码路径 - 使用uint64寄存器模拟SIMD操作

Claim（性能提升）

  • 解码速度 - >1000亿整数/秒（单核）
  • 每周期处理值 - 最高70个值/周期（8位类型）
  • 端到端查询加速 - 压缩数据扫描更快
  • 自动向量化 - 标量代码性能与显式SIMD相当

方法原理

隐含假设

  • 分析工作负载主要是顺序扫描而非随机访问
  • 关系代数操作是集合式的，不依赖元组顺序
  • 现代CPU具有宽发射能力和指令级并行性
  • 编译器能够有效自动向量化规则的数据访问模式

工作流程

原始数据 → 统一转置布局重排 → 轻量级压缩编码 → 1024位交错位打包 → 标量/SIMD解码 → 查询管道处理

1024位交错布局

在3位编码的示例中，FastLanes使用最小的适合车道宽度（8位），因此在FLMM1024字中有128个车道（1024/8=128）。逻辑上连续的3位编码值被循环分布到这些车道中。

统一转置布局

为了解决不同列的数据类型宽度不同的问题，FastLanes设计了统一转置布局，使用"04261537"顺序组织8个8×16转置块。

布局生成算法

  1. 将1024个值划分为8个8×16的块
  2. 按"04261537"顺序排列这些块
  3. 根据数据类型宽度调整处理顺序

处理顺序

  • 64位类型: 0→1→...→7
  • 32位类型: 04→15→26→37
  • 16位类型: 0426→1537
  • 8位类型: 04261537

DELTA解码算法

// 伪代码：DELTA解码算法
输入: 统一转置布局的位打包DELTA值 + 基值
输出: 解码后的值向量

1. 加载基值到SIMD寄存器
2. 对于每个SIMD车道位置i并行执行:
   - 从位打包数据中解包DELTA值
   - 执行前缀和: value[i] = base + cumulative_sum(deltas[0..i])
3. 存储解码后的值

FastLanes-RLE编码

FastLanes-RLE将传统RLE映射到DELTA编码，支持在统一转置布局中存储。

传统RLE

  • 格式: (value, length) 对
  • 需要嵌套循环解码
  • 短长度时分支预测错误率高

FastLanes-RLE

  • 字典: 运行值向量（可能包含重复）
  • 索引向量: 单调递增，新运行时增加1
  • 对索引向量应用1位DELTA编码
  • 使用统一转置布局存储

关键实验结果

位解包性能

  • 在Intel Ice Lake上达到70值/周期（8位类型）
  • Scalar_T64比原生Scalar快8倍（8位类型）
  • 编译器自动向量化效果与显式SIMD相当

DELTA解码性能

  • 统一转置布局显著加速DELTA解码
  • 在较快平台上达到>40值/周期（8位DELTA）
  • Scalar_T64从统一转置布局中的数据并行性受益

端到端查询性能

在Tectorwise向量化查询处理器中集成FastLanes，执行SELECT SUM(COL)查询，结果显示：

  • 读取FastLanes压缩数据通常使查询更快，尽管需要解压缩
  • 并行执行增加RAM瓶颈：8线程时端到端性能比未压缩快7倍
  • FastLanes将查询变快的交叉点从>4x压缩比（Scalar）移动到几乎任何数据

结论与未来工作

主要结论

  • FastLanes通过虚拟1024位SIMD寄存器和统一转置布局，在异构硬件环境下实现了高效、可移植的轻量级压缩数据解码
  • 标量代码通过自动向量化达到与显式SIMD内部函数相当的性能，提高了代码可移植性和未来兼容性
  • 统一转置布局通过打破顺序依赖，显著加速了DELTA和RLE等编码的解码性能
  • 在实际查询中，FastLanes压缩数据扫描可以比未压缩数据扫描更快，减少了内存带宽需求

局限性

  • 块大小限制：1024值的固定块大小可能对小数据集不够灵活，影响压缩率
  • 顺序敏感查询：在需要保持原始顺序的查询中，需要维护选择向量，可能带来性能开销
  • 随机访问性能：主要优化顺序扫描，随机访问模式可能无法充分发挥性能优势
  • 异常值处理：当前异常处理机制（如Patching）可能不够完善，影响极端数据分布的压缩效果
  • 硬件特性利用：虚拟1024位寄存器可能无法充分利用某些硬件的特定优化指令

未来工作方向

  • 级联轻量级压缩：研究压缩方法的堆叠组合，减少对通用压缩方法（如zstd、Snappy、LZ4）的依赖
  • GPU/TPU支持：探索在众核硬件上的高效实现
  • 完整系统集成：将FastLanes集成到完整的大数据文件格式中
  • 异常处理机制：研究更复杂的异常值处理方案
  • 压缩向量执行：研究在查询管道中利用部分压缩向量的优化机会

影响与意义

技术影响

  • 为大数据格式提供了未来证明的压缩布局
  • 解决了硬件多样性带来的技术债务问题
  • 推动了标量代码自动向量化在数据库系统中的应用

实际应用

  • 可用于Parquet、ORC等列式存储格式的优化
  • 适用于云环境中多硬件平台的数据库系统
  • 为向量化查询执行引擎提供高效的压缩数据扫描
