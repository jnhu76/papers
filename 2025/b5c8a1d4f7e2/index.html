摘要

现代操作系统内核（如Linux）使用eBPF子系统来允许用户空间扩展内核功能。为确保安全性，内核中的验证器会静态分析这些扩展；然而，其不精确的分析经常导致错误拒绝安全扩展，暴露了验证器精度和计算复杂性之间的关键张力，限制了内核可扩展性。

作者提出了一种证明引导的抽象精化技术，在保持低内核空间复杂性的同时，显著提高了验证器的精度。关键见解是将复杂推理与内核验证器解耦，同时通过形式化证明弥合差距。当遇到不确定性时，验证器启动抽象精化过程而不是拒绝扩展。

评估结果显示，该技术使验证器能够接受403个之前被错误拒绝的真实eBPF程序中的512个，为更可靠和灵活的内核扩展铺平了道路。

背景与动机

eBPF子系统的重要性

现代操作系统内核设计为高度可扩展，以适应不同的用户空间工作负载。Linux中的eBPF子系统是一个突出示例，它允许用户空间通过扩展向内核注入专门功能。这些扩展支持各种任务，包括性能分析、安全监控、文件系统管理和设备驱动程序增强。

实现定制化进程调度策略，显著提高性能和CPU利用率

安全监控

部署安全监控和策略执行框架

网络管理

实现高性能网络数据包处理和管理

eBPF验证器的作用

由于eBPF扩展在内核空间执行但由不受信任的第三方开发，内核必须严格确保其安全执行而不依赖信任假设。这种安全性由内核中的静态分析器（通常称为验证器）保证，它利用抽象解释技术。

验证器系统地探索所有潜在的控制流路径，并通过抽象语义解释每个指令，验证关键属性，如内存安全和正确内核交互。任何违规都会导致静态拒绝。通过将验证器直接嵌入内核，这种方法提供了强大的、可证明的安全保证。

问题与挑战

精度与复杂性的平衡

尽管eBPF验证器提供了强大的安全保证和扩展的高性能，但设计精确的内核验证器面临重大挑战，特别是在平衡分析精度与系统复杂性方面。提高精度通常需要纳入复杂的静态分析技术。

内核环境约束

内核环境对验证器复杂性和运行时开销施加严格约束，这与用户空间的约束明显不同。增加验证器复杂性不仅扩大了潜在攻击面，从而引入安全漏洞，还会导致运行时开销，对系统延迟产生不利影响并消耗内核资源，最终降低用户空间应用程序性能。

现有验证器的局限性

在实践中，eBPF验证器牺牲分析精度以确保低计算复杂性。为促进高效分析，验证器采用廉价的抽象域，特别是跟踪变量范围的区间域和捕获每比特值信息的三态域。这两个域都支持线性时间分析和推理。

然而，这些抽象在每次解释步骤中本质上过度近似，丢弃了关于具体计算的精确信息。因此，程序状态的抽象迅速变得粗糙，导致许多安全扩展被错误拒绝。

技术解释：
  • 区间域局限性: 区间域在算术运算后丢失精确信息，导致过度近似
  • 三态域局限性: 三态域在处理逻辑运算时精度不足
  • 信息丢失: 精确计算被局部更新的过度近似所取代
  • 变量关系: 复杂的计算和变量间关系无法充分跟踪
设计实现：
示例代码: 
0: r1 = map_lookup(...)  // r1 = ptr(size=16)
1: r2 &= 0xf             // r2 ∈ [0,15]
2: r2 <<= 1              // r2 ∈ [0,30]
3: r1 += r2              // r1.off ∈ [0,30]
4: r0 = *((u8 *)r1)      // unsafe: off+access_sz > mem_sz
问题分析:

在第2步左移后，r2的范围更新为[0,30]，但移位后的r2的最后一位必须保持为零，表明近似范围中近一半的值是不可行的。

现有改进工作的局限性

先前的研究主要集中于增强验证器的现有分析或纳入新的抽象域。例如，Vishwanathan等人通过正式确定其三态域加减运算符分析的最优性，并提出了更精确的乘法方法，提高了验证器三态域内运算符的精度。

然而，这些增强并没有克服三态域固有的基本限制。为了减轻这些约束，PREVAIL采用了Zone抽象域，它提供了比区间和三态更高的精度。但是，高级分析的集成总是引入精度-复杂性权衡。

BCF方法概述

基于设计原则，作者提出了证明引导的抽象精化技术，并将其应用于增强eBPF验证器，称为BCF（eBPF证书框架）。

技术细节：BCF工作流程
原图 2: BCF工作流程
论文原图
原图图注: BCF工作流程：证明引导的抽象精化过程
技术解释：
  • 步骤1 - 抽象解释: 验证器执行高效的抽象解释
  • 步骤2 - 精化触发: 当检测到潜在错误时启动精化
  • 步骤3 - 符号跟踪: 符号化跟踪当前分析路径以捕获程序状态的精确表示
  • 步骤4 - 精化条件: 派生精化抽象并捕获其正确性的形式条件
  • 步骤5 - 证明生成: 在用户空间使用SMT求解器生成证明
  • 步骤6 - 证明检查: 在内核空间高效验证证明
设计实现：
符号表达式:

BCF使用符号表达式跟踪程序状态，产生没有信息损失的精确表示。例如，r1的偏移量可以表示为(sym&0xf) + (0xf - (sym&0xf))，该表达式对所有sym赋值始终计算为0xf。

精化条件:

精化条件确保目标寄存器的所有可能运行时值都位于精化范围内。该条件被编码为证明义务并传输到用户空间。

工作负载委托:

推理精化条件需要复杂技术，因此被委托给用户空间。可以使用各种成熟工具，如SMT求解器和定理证明器。

证明检查机制

由于内核不信任来自用户空间的输入，因此在收到证明时执行严格的证明检查。证明是明确定义的证明规则的有序应用序列，每个规则都将其前提可靠地转换为结论。

证明检查器执行证明的线性扫描，验证每个步骤的前提和结论是否符合规定的规则。任何偏差都会导致立即拒绝证明。

BCF实现细节

BCF的实现包括对eBPF验证器的修改、内核证明检查器的集成、用户空间和内核空间之间的共享接口定义(uapi)、加载器和cvc5求解器，总计7,366行C/C++代码。

结论

作者提出了一种证明引导的抽象精化技术，以增强内核验证器的精度，同时保持其低复杂性。通过将复杂推理与内核验证器解耦，并通过形式化证明弥合差距，作者的方法显著提高了验证器的精度，同时保持了内核空间的低复杂性。

评估结果显示，该方法使验证器能够接受403个之前被错误拒绝的真实eBPF程序中的512个，为更可靠和灵活的内核扩展铺平了道路。

未来研究方向

  • 优化证明生成和检查过程以减少分析时间
  • 将方法扩展到其他内核验证场景
  • 研究如何将技术集成到现有内核开发流程中

作者的工作展示了形式化方法在实际系统验证中的潜力，为未来内核安全性和可扩展性的研究提供了重要基础。
